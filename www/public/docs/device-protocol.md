# 快速开始

本文档介绍如何将设备接入 DraARL 平台。接入前请确保您已完成以下准备工作：

- 已完成注册并审核通过
- 已获取设备密码
- 拥有有效的业余电台呼号

---

## 接入简要

![设备接入流程](../../assets/device-access-flow.svg)

**步骤 1-2：注册与审核**

在平台完成注册，填写真实的业余电台呼号。管理员审核通过后，您的账号状态将变为"已激活"。

**步骤 3：获取设备密码**

登录后进入「设备管理」页面，点击右上角「设备密码」按钮即可查看。设备密码由系统自动生成，如需更换可点击刷新按钮，刷新后原密码将立即失效。

**步骤 4：配置设备**

在您的设备客户端中填入以下认证信息：

- **用户名**：您的平台用户名
- **设备密码**：设备管理页面获取的设备准入密码
- **服务器地址**：平台提供的 UDP 服务器地址和端口

> **无输入设备**：如果您的设备没有键盘/触摸屏（如部分 ESP32 设备），可以使用「动态码绑定」功能完成首次配置，详见下方「动态码绑定流程」章节。

**步骤 5：开始通信**

设备连接成功后，会自动进行认证。认证通过后，您的呼号将由服务器自动填充。之后即可在同一群组内与其他设备进行语音和文本通信。

## 协议概述

**DraARLv1** (Dra Amateur Radio Link v1，麟链v1) 是 DraARL 平台的设备通信协议，用于设备与服务器之间的实时通信。

该协议专为业余无线电通信场景设计，采用 UDP 作为传输层协议，具有低延迟、高效率的特点。协议支持语音通联、文本消息、位置共享等功能，适用于各种客户端设备（移动端、桌面端、浏览器等）。

`UDP 传输` `大端序` `90字节固定头部` `变长负载` `Opus 16K 语音编码`

### 协议特点

- **轻量高效**：固定 90 字节头部，解析快速，适合嵌入式设备
- **安全认证**：设备密码 + 阶梯封禁机制，防止暴力破解
- **语音优化**：统一采用 Opus 16kHz 编码，音质与带宽平衡
- **群组通信**：支持多群组切换，同一群组内设备可互相通信
- **服务器互联**：支持多服务器部署，跨服务器语音转发

> 本协议为业余无线电通信设计，请遵守当地无线电管理法规。

## 报文结构

协议报文由固定头部（90字节）和变长数据区组成。所有多字节字段使用大端序（Big-Endian）。

```text
+--------+--------+--------+--------+--------+--------+--------+--------+
| 0-3    | 4-5    | 6-37   | 38-47  | 48     | 49     | 50     | 51-53  |
| Ver    | Length |Username|DevPass | Type   |DevModel| SSID   | DMRID  |
| 4B     | 2B     | 32B    | 10B    | 1B     | 1B     | 1B     | 3B     |
+--------+--------+--------+--------+--------+--------+--------+--------+
| 54-85  | 86-89  | 90+                                              |
|CallSign| Rsv    | DATA                                             |
| 32B    | 4B     | 变长                                             |
+--------+--------+--------+-----------------------------------------+

固定头部：90 字节
最小报文：90 字节
最大报文：800 字节（UDP 限制）
```

### 字段说明

| 偏移 | 长度 | 字段名 | 类型 | 说明 |
|------|------|--------|------|------|
| 0 | 4B | Version | string | 协议版本标识，固定为 "DraA" |
| 4 | 2B | Length | uint16 BE | 报文总长度（包含头部和数据） |
| 6 | 32B | Username | string | 用户名，UTF-8 编码，\0 填充 |
| 38 | 10B | DevicePassword | string | 设备准入密码，ASCII 字母数字 |
| 48 | 1B | Type | byte | 数据包类型 |
| 49 | 1B | DevModel | byte | 设备型号 |
| 50 | 1B | SSID | byte | 设备子号 (0-255) |
| 51 | 3B | DMRID | uint24 BE | DMR ID |
| 54 | 32B | CallSign | string | 业余电台呼号，服务器填充 |
| 86 | 4B | Reserved | - | 保留字段，填 0 |
| 90 | 变长 | DATA | []byte | 负载数据 |

## Config 包协议

> **TypeConfig (Type=3)** 仅用于 UDP 普通设备（SSID 1-99, 106-235）的配置同步，不适用于幽灵设备（App/PC 客户端）。

### 包类型

| DATA[0] | 方向 | 说明 |
|---------|------|------|
| 0x01 | 服务端 → 设备 | 查询配置请求 |
| 0x02 | 双向 | 配置下发/上报（支持动态下发部分配置项）|
| 0x03 | 服务端 → 设备 | 时间同步 |

### TLV 数据格式

当 DATA[0] = 0x02 时，DATA 区域格式：

```text
DATA[0] = 0x02 (配置标识)
DATA[1] = N (配置项数量)
DATA[2...] = TLV 列表

TLV 格式：
┌──────────┬──────────┬─────────────────┐
│ Type (1B)│ Len (1B) │ Value (N bytes) │
└──────────┴──────────┴─────────────────┘
```

当 DATA[0] = 0x03 时，DATA 区域格式：

```text
DATA[0]    = 0x03 (时间同步标识)
DATA[1]    = 保留字节 (当前实现固定为 0x00)
DATA[2:10] = Unix 时间戳 (8字节，big-endian int64，毫秒)
```

### 配置项 Type 定义

| Type | 配置键 | 长度 | 格式 | 说明 |
|------|--------|------|------|------|
| 0x01 | rx_freq | 8 | big-endian uint64 | 接收频率 (Hz) |
| 0x02 | tx_freq | 8 | big-endian uint64 | 发射频率 (Hz) |
| 0x03 | rx_ctcss | 4 | big-endian float32 | 接收亚音 (Hz, 0=关闭) |
| 0x04 | tx_ctcss | 4 | big-endian float32 | 发射亚音 (Hz, 0=关闭) |
| 0x05 | sql_level | 1 | uint8 | 静噪等级 (0-8) |
| 0x06 | power_level | 1 | uint8 | 功率等级 (1=低, 3=高；历史值 2 兼容映射为高) |
| 0x07 | tx_bandwidth | 1 | uint8 | 发射带宽 (1=窄带, 2=宽带) |
| 0x08 | rx_tone_mode | 1 | uint8 | 接收亚音类型 (0=OFF, 1=CTCSS, 2=CDCSS_N, 3=CDCSS_I) |
| 0x09 | rx_tone_value | 8 | ASCII | 接收亚音值（如 `88.5`、`023`） |
| 0x0A | tx_tone_mode | 1 | uint8 | 发射亚音类型 (0=OFF, 1=CTCSS, 2=CDCSS_N, 3=CDCSS_I) |
| 0x0B | tx_tone_value | 8 | ASCII | 发射亚音值（如 `88.5`、`023`） |
| 0x10 | timestamp | 8 | big-endian int64 | Unix 时间戳 (毫秒) |

### 频率配置约定

- `tx_freq / rx_freq` 保持手动输入，单位为 Hz。
- 当前频率设置卡片按 `devmodel + cardid` 选择实现，本轮默认卡片为 `sa818-radio-v1`。
- 亚音采用双层模型：
  - `OFF`：值固定为 `0`
  - `CTCSS`：标准 38 个模拟亚音值
  - `CDCSS_N / CDCSS_I`：SA818 数字亚音码表，正/反向极性分开
- 服务端保留 `rx_ctcss / tx_ctcss` 作为镜像字段：
  - 模拟亚音时回填实际 Hz 值
  - 数字亚音或解析失败时旧字段回退为 `0`（无亚音）
- 数字亚音完整下发使用 `rx_tone_* / tx_tone_*`，设备侧可按 SA818 实际参数格式组合为 `023N / 023I` 等表达。

### 示例

**下发单个配置项（接收频率 439.500MHz）**：

```text
DATA = [0x02, 0x01, 0x01, 0x08, 0x00, 0x00, 0x00, 0x00, 0x1A, 0x32, 0x3C, 0xE0]
         │     │     │     │     └─────────────────────────────────────────────┘
         │     │     │     │                       439500000 Hz (0x1A323CE0)
         │     │     │     └─ 长度 8
         │     │     └─ Type=rx_freq
         │     └─ 配置项数量 1
         └─ 配置下发标识
```

**设备上报全部配置**：

```text
DATA = [0x02, 0x0B, TLV1, TLV2, TLV3, TLV4, TLV5, TLV6, TLV7, TLV8, TLV9, TLV10, TLV11]
              └─ 典型全量上报为 11 个配置项
```

**完整数字亚音下发（示例，按 Type 升序）**：

```text
DATA = [
  0x02, 0x0B,
  0x01, 0x08, 0x00, 0x00, 0x00, 0x00, 0x1A, 0x32, 0x3C, 0xE0,                    // rx_freq = 439500000
  0x02, 0x08, 0x00, 0x00, 0x00, 0x00, 0x19, 0xB8, 0x2A, 0xE0,                    // tx_freq = 431500000
  0x03, 0x04, 0x00, 0x00, 0x00, 0x00,                                              // rx_ctcss = 0
  0x04, 0x04, 0x00, 0x00, 0x00, 0x00,                                              // tx_ctcss = 0
  0x05, 0x01, 0x05,                                                                  // sql_level = 5
  0x06, 0x01, 0x03,                                                                  // power_level = 3 (高)
  0x07, 0x01, 0x02,                                                                  // tx_bandwidth = 2 (宽)
  0x08, 0x01, 0x02,                                                                  // rx_tone_mode = CDCSS_N
  0x09, 0x08, 0x30, 0x32, 0x33, 0x00, 0x00, 0x00, 0x00, 0x00,                      // rx_tone_value = "023"
  0x0A, 0x01, 0x03,                                                                  // tx_tone_mode = CDCSS_I
  0x0B, 0x08, 0x34, 0x33, 0x31, 0x00, 0x00, 0x00, 0x00, 0x00                       // tx_tone_value = "431"
]
```

> 说明：服务端编码时遍历 map，TLV 实际顺序可变；设备应按 Type 解析，不应依赖顺序。

### 同步流程

**设备上线同步**：

```text
设备上线 (认证成功)
    │
    ├─ 数据库有配置记录
    │   └─ 服务端发送 Config(0x02 + TLV) 下发配置
    │
    └─ 数据库无配置记录
        └─ 服务端发送 Config(0x01) 查询请求
            │
            └─ 设备上报 Config(0x02 + TLV)
                │
                └─ 服务端解析存储到数据库
```

**控制台修改同步**：

```text
用户通过控制台修改配置
    │
    ├─ 保存变更到数据库
    │
    └─ 设备在线？
        ├─ 是 → 发送 Config(0x02 + 变更项TLV) 动态下发
        └─ 否 → 等待设备上线时同步
```

**时间同步**：

服务端会在普通 UDP 设备上线完成配置同步后发送一次时间同步包；设备上报配置后，服务端也会再发送一次时间同步包作为 ACK。

```text
DATA = [0x03, 0x00, ts(8 bytes)]
        │     │     └───────────── Unix 时间戳（毫秒，大端序）
        │     └─ 保留字节（当前实现固定为 0）
        └─ 时间同步标识
```

## Opus 语音格式

> **TypeOpus16K (Type=5)** 是平台的标准语音编码格式，采用 Opus 编解码器，专为业余无线电通信优化。

### 编码参数

| 参数 | 值 | 说明 |
|------|------|------|
| 采样率 | 16000 Hz | 16kHz 宽带语音 |
| 声道数 | 1 | 单声道 |
| 帧时长 | 60 ms | 每个 Opus 帧 60ms |
| 每帧采样数 | 960 | 16000 × 0.06 |
| 比特率 | 16-32 kbps | VOIP 模式 |

### WebSocket 客户端合并帧格式

为优化弱网性能，WebSocket 客户端采用 **2 帧合并发送** 策略：每 120ms 发送一个数据包，每个数据包包含 2 个 60ms 的 Opus 帧。

```text
+-------------+-------------+-------------+-------------+
| Frame1 Len  | Frame1 Data | Frame2 Len  | Frame2 Data |
| 2B          | N bytes     | 2B          | M bytes     |
+-------------+-------------+-------------+-------------+

Frame Len: 帧长度（大端序 uint16）
Frame Data: Opus 帧数据（60ms）
```

### 性能优化效果

| 项目 | 优化前 | 优化后 |
|------|--------|--------|
| 帧时长 | 20ms | 60ms |
| 发送间隔 | 20ms | 120ms |
| 每秒发包数 | 50 | ~8.3 |
| 单包大小 | 110-170B | 330-570B |
| 头部开销占比 | 50-80% | ~25-35% |
| 额外延迟 | - | +100ms |

> **解码兼容性**
>
> 接收端应同时支持：
> - **合并帧格式**：按长度前缀解析多个 Opus 帧
> - **单帧格式**：兼容无长度前缀的单帧数据（用于 UDP 设备直发）
>
> 判断逻辑：从起始位置按 `2B 长度 + 帧数据` 循环解析；若首个长度非法（`0`、`>1000`、或越界）则回退单帧；若已解析到至少 1 帧，后续非法长度会停止继续解析并保留已解析帧。

## 安全限制

UDP 服务端实施以下安全策略，超出限制的数据包将被静默丢弃：

| 限制项 | 阈值 | 说明 |
|--------|------|------|
| 最大包体 | 800 字节 | 含 90 字节头部，防止异常大包攻击 |
| 包速率 | 150 包/秒 | 按 IP+Port 维度限速，兼顾 FRP 隧道共享 IP 场景 |

> 正常语音通信（合并帧 ~8.3 包/秒，单帧 ~16.7 包/秒）远低于限速阈值，不会受到影响。
> 150 包/秒的阈值设计考虑了 FRP/NAT 共享 IP、网络抖动突发场景、以及大包架构下丢包的严重影响。

## 设备认证流程

平台支持三种认证方式，根据设备类型和场景自动选择：

### 认证方式

| 设备类型 | SSID 范围 | 认证方式 | 说明 |
|----------|----------|----------|------|
| **普通设备** | 1-99, 106-235 | 设备密码 | ESP32、互联设备、对讲桥接器等嵌入式/桥接设备 |
| **幽灵设备** | 101-105 | JWT Token | App/PC 客户端 (Android/iOS/Windows/macOS/Web) |
| **普通设备（无输入）** | 1-99, 106-235 | 动态码绑定 | 新设备首次绑定，适用于无键盘/触摸屏的嵌入式设备 |

**普通设备**（如 ESP32 链路台/手咪、互联设备、对讲桥接器）性能有限，不适合进行 JWT 解析，使用传统的设备密码认证。其中：
- 带有键盘/触摸屏的设备：可直接输入用户名和设备密码
- 无输入能力的设备：通过动态码绑定流程完成首次配置

**幽灵设备**（App/PC 客户端）使用 JWT 认证，可以复用现有登录接口，无需预共享密钥。

### 1. 普通设备认证（设备密码）

设备通过发送心跳包（Type=2）进行认证。认证成功后，服务器会在响应中填充用户的呼号。

![认证流程时序图](../../assets/auth-sequence.svg)

> **安全机制**
>
> 为防止暴力破解，平台采用阶梯封禁机制：
> - 当前实现中，连续失败第 4 次触发首轮封禁：10 秒
> - 后续继续失败依次升级：30 秒、60 秒、300 秒（递增）
>
> 封禁 Key：IP + Username

### 2. 幽灵设备认证（JWT Token）

幽灵设备（App/PC 客户端）为平台官方提供的标准客户端，通过 JWT Token 进行认证。

### 3. 动态码绑定流程（无输入设备）

适用于没有键盘/触摸屏的嵌入式设备（如带屏幕但不支持输入的 ESP32 设备）。设备通过 HTTP 接口完成与用户账号的绑定。

![动态码绑定流程](../../assets/dynamic-bind-sequence.svg)

#### 设备端 HTTP 接口

| 接口 | 方法 | 认证 | 说明 |
|------|------|------|------|
| `/api/device/pre-check` | POST | 无 | 检查设备存储的账号密码是否有效 |
| `/api/device/request-code` | POST | 无 | 请求生成 6 位动态码 |
| `/api/device/confirm-bind` | POST | 无 | 设备轮询获取绑定状态和配置 |

#### 步骤详解

**步骤 1：预检查**

设备上电后，使用本地存储的账号密码（如果有）进行检查：

请求：
```json
POST /api/device/pre-check
{
    "mac": "AA:BB:CC:DD:EE:FF",
    "username": "user1",
    "device_password": "abc12345"
}
```

响应：
- 认证成功：`{ "status": "authenticated", "call_sign": "BG1AAA" }`，设备可直接开始 UDP 认证
- 认证失败/无存储：`{ "status": "need_bind", "message": "请使用动态码绑定设备" }`，进入动态码绑定流程

**步骤 2：请求动态码**

设备请求生成动态码：

请求：
```json
POST /api/device/request-code
{
    "mac": "AA:BB:CC:DD:EE:FF"
}
```

响应：
```json
{
    "dynamic_code": "123456",
    "expires_in": 60
}
```

设备在屏幕/LED 上显示 6 位动态码。

**步骤 3：用户 Web 操作**

用户登录 Web 平台，在「设备管理」页面点击「动态码绑定」，输入设备显示的动态码并配置 SSID。

**步骤 4：确认绑定**

设备轮询此接口获取配置：

请求：
```json
POST /api/device/confirm-bind
{
    "mac": "AA:BB:CC:DD:EE:FF"
}
```

响应：
- 等待中：`{ "status": "waiting", "message": "等待用户完成绑定" }`
- 绑定完成：
```json
{
    "status": "ready",
    "username": "user1",
    "device_password": "abc12345",
    "ssid": 1,
    "dmr_id": 4601234
}
```

设备获取到配置后，保存在本地存储中，然后开始 UDP 认证流程。

#### 安全机制

- **动态码单次使用**：绑定成功后动态码立即失效
- **有效期限制**：动态码 60 秒过期，绑定状态 10 分钟过期
- **限速保护**：各接口均有独立的限速策略
- **账号状态校验**：用户必须已审核通过才能绑定设备

## 字符编码规则

各字段使用不同的编码规则，请确保设备端正确处理：

| 字段 | 编码 | 字符集 | 说明 |
|------|------|--------|------|
| Username | UTF-8 | 字母、数字、下划线 | 用户名 |
| DevicePassword | ASCII | 大小写字母、数字 | 设备准入密码，6-10 位 |
| CallSign | ASCII | 大写字母、数字 | 业余电台呼号 |
| TextMessage | UTF-8 | 任意 Unicode | 文本消息 |

## 常见问题

### Q: 设备显示认证失败怎么办？

请检查：
1. 用户名是否正确
2. 设备密码是否正确（注意大小写）
3. 账号是否已审核通过
4. 是否因多次失败被封禁（等待封禁时间过期）

### Q: 设备多久没有数据会被判定离线？

服务器检测到设备 20 秒内无任何数据包（包括心跳和语音数据），将判定设备离线。建议设备端每 10-15 秒发送一次心跳包保持在线状态。

### Q: 支持哪些语音编码格式？

平台统一使用 Opus 16kHz 编码格式。这是目前业余无线电网络中音质与带宽平衡的最佳选择。

### Q: 如何切换通信群组？

登录平台后，在「设备管理」页面选择要操作的设备，点击「切换群组」按钮，选择目标群组后保存即可。设备会在下次心跳时获取新的群组配置。
